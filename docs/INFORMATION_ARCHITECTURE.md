# Purpose

This document defines the information architecture of Portfolio 2.0.

It does not define:

* Frontend architecture
* Backend architecture
* UI design
* Navigation
* Visual design
* Page layouts
* Components

Its purpose is to define:

* Information domains
* Relationships between domains
* Information hierarchy
* Discovery model
* Visitor understanding model

The architecture should remain valid even if the implementation changes.

---

# Core Principle

Portfolio 2.0 is not organized around projects.

Portfolio 2.0 is not organized around technologies.

Portfolio 2.0 is not organized around visitor categories.

Portfolio 2.0 is organized around the creator.

The creator is the protagonist.

Everything else exists as supporting evidence.

---

# Information Hierarchy

The portfolio follows the following hierarchy:

```text
Atharva
│
├── Identity
│
├── Capabilities
│
├── Evidence
│
├── Interaction
│
└── Contact
```

All information within the portfolio should belong to one of these domains.

---

# Domain 1 — Identity

## Purpose

Answer:

> Who is Atharva?

before answering:

> What has Atharva built?

Identity should be understood before evidence.

---

## Contains

### Professional Identity

Current role.

Current focus.

Current expertise.

Current direction.

---

### Philosophy

How the creator thinks.

How the creator approaches engineering.

How the creator approaches design.

How the creator approaches products.

How the creator approaches creativity.

---

### Principles

The beliefs that guide work.

Examples:

* Simplicity
* Craftsmanship
* Systems Thinking
* User-Centered Design
* Engineering Excellence
* Attention To Detail

---

### Current State

Identity should represent the present.

The portfolio is not intended to be a biography.

Historical narrative should be minimized.

The focus is:

> Who is Atharva today?

# Virtual Me Placement

Virtual Me is not the portfolio.

Virtual Me is not a supporting utility.

Virtual Me is one of the major pillars of the experience.

It should feel important without becoming mandatory.

---

## Important Rule

Visitors should be able to understand the creator without using Virtual Me.

Virtual Me should deepen understanding.

Not provide required understanding.

---

# Discovery Model

Portfolio 2.0 follows a discoverability model rather than a guided journey model.

Visitors should feel free to explore.

However, exploration should be intentionally supported.

The architecture should provide multiple entry points into the same underlying information.

---

# Discovery Principles

## Principle 1

Information should be easy to find.

The portfolio is not a puzzle.

The portfolio is not a treasure hunt.

---

## Principle 2

Visitors should never feel lost.

---

## Principle 3

Different visitors may explore different paths.

All paths should ultimately reveal the same creator.

---

## Principle 4

Depth should be optional.

Understanding should not require extensive exploration.

Deeper exploration should reward curiosity.

---

# First Impression Requirements

Within the first few seconds visitors should understand:

1. This is a portfolio.
2. This is different from most portfolios.
3. This feels premium.
4. There is an AI layer.
5. There is more depth available.

The first impression should create intrigue.

Not confusion.

---

# Exclusions

The portfolio is intentionally not designed around:

* Personal biography
* Chronological life story
* Career timeline storytelling
* Hidden information
* Artificial complexity
* Visitor segmentation

The focus remains:

Present identity.

Present capability.

Demonstrated proof.

Meaningful interaction.
